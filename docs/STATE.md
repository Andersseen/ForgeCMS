## TL;DR

The monorepo foundation is done and CI is green. The schema DSL, validation, runtime orchestrator,
CRUD HTTP handlers, and several real adapters (LibSQL, Cloudflare D1/R2) **exist and are tested**.
As of 2026-07-08, `apps/www`'s routes actually delegate to `@forge-cms/runtime` (spec 007), writes
are protected by a real signed-token auth provider with a login page (spec 006), an e2e suite covers
the CRUD+auth flow (PLAN.md P1-4), and the real admin UI (layout, document list, schema-driven form)
lives in `@forge-cms/admin` and is consumed by `apps/www`, not duplicated in it (spec 008). Remaining
gap: nothing is published to npm yet (PLAN.md P3-2, blocked on nothing now except an actual publish
step requiring registry credentials), and the demo has never been wired to real Cloudflare bindings
end-to-end in production (D1 persistence, spec 005, is done and locally-verified but its production
deploy is unconfirmed from this environment).

## API surface (implemented in apps/www server routes)

- `GET /api/v1/[collection]` — list (supports `limit`, `offset`, and equality `where` filters via any
  other query param, coerced to the field's type); `POST /api/v1/[collection]` — create (validated,
  **requires auth**)
- `GET /api/v1/[collection]/[id]` — read; `PUT`/`DELETE /api/v1/[collection]/[id]` — update/delete
  (validated, **require auth**)
- `GET /api/v1/collections` — collection metadata (+ `fieldDefinitions`, added 2026-07-07)
- `GET /api/auth/me` — current user; `POST /api/auth/login` — `{ email, password }` →
  `{ token, user }` (added 2026-07-08, spec 006)
- `GET /api/status` — adapter/API health info

✅ **Routes now delegate to `@forge-cms/runtime`'s handlers** (spec 007, 2026-07-08) — matches
ARCHITECTURE.md's "thin wrappers" claim; known issue #1 resolved. Success envelope: list
`{ data, meta }`, item `{ data }` (create → `201`), delete `204`. Error envelope matches
ARCHITECTURE.md: `{ error: string, details?: ValidationError[] }`.

## Infrastructure

- CI (GitHub Actions): `ci.yml` (lint + typecheck + test + build on push/PR to main); `e2e.yml`
  (Playwright, separate workflow); `deploy-cloudflare.yml` (checks job — format/lint/typecheck/test/e2e —
  then deploys `apps/www/dist/analog/public` to Cloudflare Pages; needs `CLOUDFLARE_API_TOKEN`,
  `CLOUDFLARE_ACCOUNT_ID` secrets). The previously-duplicate ungated `deploy.yml` was removed (PLAN.md
  P1-1 / spec 002) — it raced `deploy-cloudflare.yml` on every push with no test gate.
- Changesets configured; **no package published yet** (all 0.0.0).
- `wrangler.toml`: Pages config, output `apps/www/dist/analog/public` (Nitro `cloudflare-pages`
  preset — includes `_worker.js`, the compiled API server), `nodejs_compat`.

## Known issues / debt (verified 2026-07-07, updated 2026-07-08)

1. ~~None of `apps/www`'s 5 HTTP routes delegate to `@forge-cms/runtime`'s handlers~~ — **fixed
   2026-07-08, spec 007.** All 5 CRUD routes now build an `ApiContext` and call the matching
   `handle*` function; filtering, validation, and the error envelope all now match
   ARCHITECTURE.md and `@forge-cms/runtime`'s own tests. One pre-existing bug surfaced (not a
   regression — the old app-local route had byte-identical logic): `handleUpdate`'s partial-validation
   filter treats a required field missing from a partial PUT body as an error even when that field
   already exists on the stored record and isn't part of the update. Doesn't affect the real
   schema-driven edit form (it always sends the full document), only a true partial PUT — left
   unfixed, out of spec 007's scope.
2. ~~`@forge-cms/admin` is dead code~~ — **fixed 2026-07-08, spec 008.** The real layout/document-list/
   schema-driven-form components now live in the package; `apps/www` consumes them.
3. `_routes.json`/`_headers`/`_redirects` (Nitro's Cloudflare Pages preset output) land in
   `apps/www/dist/analog/` instead of inside `apps/www/dist/analog/public/`, so they're not included
   in the deploy — cosmetic only (the worker itself already serves static assets via `env.ASSETS`
   regardless), noted as a non-goal follow-up in spec 002.

2026-07-08 — there's no standing task-by-task plan file anymore; new work should get its own spec in
`docs/specs/` per [SDD.md](SDD.md) rather than a wishlist doc. Candidates for a next round, in rough

   in the runtime, a real multipart upload flow for storage, `where` operators beyond equality.
