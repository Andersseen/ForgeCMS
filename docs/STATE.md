## TL;DR

The monorepo foundation is done and CI is green. The schema DSL, validation, runtime orchestrator,
CRUD HTTP handlers, and several real adapters (LibSQL, Cloudflare D1/R2) **exist and are tested**.
As of 2026-07-08, `apps/www`'s routes actually delegate to `@forge-cms/runtime` (spec 007), writes
are protected by a real signed-token auth provider with a login page (spec 006), an e2e suite covers
the CRUD+auth flow (PLAN.md P1-4), and the real admin UI (layout, document list, schema-driven form)
lives in `@forge-cms/admin` and is consumed by `apps/www`, not duplicated in it (spec 008). As of
2026-07-20, spec 009 ships real users-collection auth: `UsersCollectionAuthAdapter` stores users
in the configured database with PBKDF2 password hashing, `/api/auth/users` CRUD is protected by
token auth, and the `/admin/users` page can create, edit, and delete users — covered by a dedicated
Playwright e2e test. Spec 010 (also 2026-07-20) adds role-based access control: admins manage users
and have full content access, editors can create/edit/delete documents, and viewers are read-only
across the backend and admin UI. **As of 2026-07-22, Phase 1 of [ROADMAP.md](ROADMAP.md) is done** — the four structural changes that
separate a typed CRUD layer from a CMS: a **Local API** on `ForgeCmsRuntime` (spec 019) with the HTTP
handlers reduced to transport, **access control as functions** returning row-level query constraints
(spec 020), the **complete hook pipeline** including field hooks (spec 021), and **composite fields**
`group`/`array`/`blocks` (spec 022). Spec 018 also fixed the D1 `passwordHash` bug that broke auth on
the only production path, and gave lists real pagination metadata.

Remaining gap: nothing is published to npm yet (ROADMAP Phase 0.3,
blocked on nothing now except an actual publish step requiring registry credentials), and the demo
has never been wired to real Cloudflare bindings end-to-end in production (D1 persistence, spec 005,
is done and locally-verified but its production deploy is unconfirmed from this environment).

## API surface (implemented in apps/www server routes)

- `GET /api/v1/[collection]` — list (supports `limit`, `offset`, `where` filters via any other query
  param — bare `field=value` for equality or `field[op]=value` for
  `ne`/`gt`/`gte`/`lt`/`lte`/`in`(comma-separated)/`contains`, coerced to the field's type, **spec 011,
  2026-07-20** — plus single-field `sort`/`order` (`asc`/`desc`), `depth=1` to populate `relation`
  fields with the actual related record instead of a bare id (**spec 012**), and — on collections with
  `drafts: true` — `status=draft|published|all` to control which documents are visible, gated on
  authentication (**spec 017, 2026-07-20**)); `POST /api/v1/[collection]` — create (validated, **requires
  admin or editor role**; accepts `multipart/form-data` on `upload: true` collections, **spec 016**)
- `GET /api/v1/[collection]/[id]` — read (also supports `depth=1`; 404s a `draft` document for
  unauthenticated requests on a `drafts: true` collection, spec 017); `PUT`/`DELETE
/api/v1/[collection]/[id]` — update/delete (validated, **require admin or editor role**)
- `GET /api/v1/collections` — collection metadata (+ `fieldDefinitions`, added 2026-07-07)
- `GET /api/auth/me` — current user; `POST /api/auth/login` — `{ email, password }` →
  `{ token, user }` (added 2026-07-08, spec 006)
- `GET /api/auth/users` — list users; `POST /api/auth/users` — create user;
  `PUT /api/auth/users/:id` — update user; `DELETE /api/auth/users/:id` — delete user
  (added 2026-07-20, spec 009; **admin-only as of spec 010**)
- `GET /api/status` — adapter/API health info

✅ **Routes now delegate to `@forge-cms/runtime`'s handlers** (spec 007, 2026-07-08) — matches
ARCHITECTURE.md's "thin wrappers" claim; known issue #1 resolved. Success envelope: list
`{ data, meta }`, item `{ data }` (create → `201`), delete `204`. Error envelope matches
ARCHITECTURE.md: `{ error: string, details?: ValidationError[] }`.

## Infrastructure

- CI/CD (GitHub Actions): **consolidated into a single pipeline `ci.yml` as of 2026-07-24.** One
  `checks` job (format:check + lint + typecheck + test + build + Playwright e2e) runs on every push/PR
  to main; a `deploy` job `needs: checks` and is gated `if: github.event_name == 'push' && github.ref ==
'refs/heads/main'`, so PRs and forks never deploy. It builds `apps/www`, verifies
  `apps/www/dist/analog/public/index.html`, then `wrangler pages deploy`s to Cloudflare Pages
  (`--project-name=forge-cms`); needs `CLOUDFLARE_API_TOKEN` + `CLOUDFLARE_ACCOUNT_ID` secrets and uses
  a `production` environment (url `https://forge-cms.pages.dev`). The old separate `e2e.yml` and
  `deploy-cloudflare.yml` were removed — before, every push to main triggered three overlapping
  workflow runs; now it's exactly one. **Cloudflare is the only deploy target.** (Earlier, the
  duplicate ungated `deploy.yml` had already been removed per spec 002.)
- Changesets configured; **no package published yet** (all 0.0.0).
- `wrangler.toml`: Pages config, output `apps/www/dist/analog/public` (Nitro `cloudflare-pages`
  preset — includes `_worker.js`, the compiled API server), `nodejs_compat`.

## Known issues / debt (verified 2026-07-22)

1. ~~`handleUpdate` partial-validation bug~~ — **fixed 2026-07-19.** Required fields already present on
   the stored record are no longer required in a partial PUT body.
2. ~~`@forge-cms/admin` is dead code~~ — **fixed 2026-07-08, spec 008.** The real layout/document-list/
   schema-driven-form components now live in the package; `apps/www` consumes them.

   `apps/www/dist/analog/` instead of inside `apps/www/dist/analog/public/`, so they're not included
   in the deploy — cosmetic only (the worker itself already serves static assets via `env.ASSETS`
   regardless), noted as a non-goal follow-up in spec 002.
4. `@forge-cms/cloudflare` previously depended on `@forge-cms/runtime` — **fixed 2026-07-19.**
5. `README.md` and root `CHANGELOG.md` were outdated — **fixed 2026-07-19.**
6. D1 `findMany` did not coerce `where` values before binding — **fixed 2026-07-19.**
7. `/api/auth/me` rebuilt the request manually — **fixed 2026-07-19**, now uses `toWebRequest`.
8. `toWebRequest(event)` consumed the request body before `readBody(event)` in `POST`/`PUT` auth routes,
   causing those routes to hang — **fixed 2026-07-20** by building a headers-only `Request` for auth
   validation after the body is read.
9. ~~Role-based access control~~ — **fixed 2026-07-20, spec 010.** Roles are now enforced in the
   backend (`/api/auth/users` admin-only, `/api/v1/*` writes restricted to admin/editor) and the
   admin UI hides write actions and the Users page for unauthorized roles.
10. ~~Production-only crash on every `/admin` page: `Error: JIT compiler unavailable`~~ — **fixed
    2026-07-20.** Two independent causes, both fixed inside this repo without touching any external
    package: - `@forge-cms/admin` and third-party libraries like `@voltui/components` build with
    `angularCompilerOptions.compilationMode: "partial"` (the correct choice for a publishable
    library — it isn't pinned to one exact Angular compiler version). Partial compilation ships
    `ɵɵngDeclareComponent`-style calls that require the consuming app's build to run the **Angular
    linker**. `apps/www`'s Vite/Analog.js pipeline had no linker step, so those declarations reached
    production unlinked; AOT prod builds tree-shake `@angular/compiler` out entirely, so Angular's
    runtime JIT fallback threw. Fixed by adding a real Angular linker to `apps/www`'s Vite build:
    `apps/www/vite-plugins/angular-linker.ts` runs `@angular/compiler-cli/linker/babel` (the
    officially documented way to integrate the linker outside the Angular CLI) over any dependency
    file that `@angular/compiler-cli/linker`'s `needsLinking` flags — this fixes every partial-Ivy
    dependency in the graph, present and future, without changing any library's compilation mode. - Separately, `@forge-cms/angular` built with plain `tsc`, not `ngc`, so `CmsApiService`'s
    `@Injectable({ providedIn: 'root' })` was never processed by Angular's compiler at all — it
    shipped as a raw runtime decorator with no static `ɵprov`, which also needs JIT (a different
    failure mode from the linking issue above, same underlying "AOT prod builds have no
    `@angular/compiler`" cause). Fixed by switching its build to `ngc` with
    `compilationMode: "partial"` (consistent with `@forge-cms/admin`); the same linker plugin
    resolves it at `apps/www`'s build time. - Verified with a real browser (Playwright) against the actual `wrangler pages dev`-served
    production build: landing, login (rendering real `@voltui/components`), and `/admin/dashboard`
    all load with zero console errors. - ~~Found but not fixed: local D1 login returns `500 D1_ERROR: table users has no column named
passwordHash`~~ — **fixed 2026-07-22, spec 018.** `@forge-cms/auth` now exports
    `withAuthFields()`, which merges the columns the auth adapter writes into the `users`
    collection definition, so schema generation and additive migration cover them. A regression
    test asserts the generated SQL contains `"passwordHash" TEXT`.

11. **`meta.count` is the page length, not the total** — **fixed 2026-07-22, spec 018.**
    `DatabaseAdapter.count` takes a `where`, and list responses carry `totalDocs`/`page`/
    `totalPages`/`hasNextPage`/`hasPrevPage`.
12. **The admin relation field is a bare text input** where you paste a document id by hand, and
    `richtext`/`upload` fall back to a textarea/text input. Composite fields render properly
    (spec 022) but scalar widgets are still primitive — ROADMAP item 032.
13. **No versions/revisions.** Spec 017 shipped draft/published status only: no history, no diff, no
    restore, no autosave — ROADMAP item 024.
14. **No globals** (singleton documents). `apps/www` fakes one with a `site_config` collection —
    ROADMAP item 023.
15. **No localisation.** Changes the storage shape, so it gets more expensive the longer real data
    exists — ROADMAP item 025.
16. **Deleting a document does not delete its storage object**, and relations are not
    referentially checked on delete (a deleted author leaves posts pointing at a dead id). Spec 021's
    `beforeDelete`/`afterDelete` hooks now make both expressible in userland, but neither is built in
    — ROADMAP item 027.

## What's next

Work is planned in [ROADMAP.md](ROADMAP.md), which sequences the remaining gaps by cost-of-delay.
Each numbered item there gets its own spec in `docs/specs/` when picked up, per [SDD.md](SDD.md).

Status as of 2026-07-22:

- **Phase 0 — Unblock:** 0.1 (D1 auth schema) and 0.2 (pagination metadata) done via spec 018.
  **0.3 (publish `0.1.0` to npm) is still open** and is the single highest-value remaining item —
  it needs registry credentials, so it is a human step: `pnpm changeset version && pnpm changeset publish`.
- **Phase 1 — Structural core: done.** Specs 019 (Local API), 020 (access as functions),
  021 (full hook pipeline), 022 (composite fields).
- **Phase 2 — Content model:** globals, versions/revisions, localisation, query completeness
  (nested and/or, multi-field sort), referential integrity. Not started.
- **Phase 3 — Auth & DX:** real auth (cookies/refresh/password reset/API keys), an email adapter,
  a config + plugin system, a CLI. Not started.
- **Phase 4 — Admin UI:** real field widgets (richtext editor, media picker, searchable relation
  picker), list-view pagination/sort/filters, conditional fields, live preview. Not started.
- **Phase 5 — The Angular/Analog moat:** signals-based client resources, an Analog integration
  package, collection types reaching the client without codegen. Not started — but per ROADMAP's
  thesis these rank in strategic value immediately after Phase 1, ahead of most of Phases 2–4,
  because they are the reason to choose ForgeCMS over an existing CMS at all.

Also unconfirmed (unchanged): the production deploy has never been verified end-to-end against real
Cloudflare bindings — `curl https://<deployed>/api/status` after the next merge to main should show
`"name": "d1"` and the real auth adapter.
